# YuhanBakery (小龙虾烘焙坊) 项目总结报告

## 1. 项目概览

### 1.1 项目背景与研发目的
随着全渠道零售模式（O2O）的成熟，烘焙行业传统线下单店面临获客半径有限、配送履约繁琐、营销方式单一等诸多痛点。为了推动数字化转型，**YuhanBakery (小龙虾烘焙坊)** 项目立项。
本项目旨在打造一个具备商户管控、骑手配送和顾客点单全链路的高性能线上新零售系统，整合“顾客领券加购 -> 门店发货调度 -> 骑手抢单配送与分账提现”的完整电商 O2O 闭环，实现数字化自主经营。

### 1.2 核心建设目标
*   **顾客体验升级**：构建操作丝滑的移动商城，引入自动化营销工具（优惠券）和基于 DeepSeek 的流式 AI 智能客服。
*   **物流履约打通**：为骑手自建高可读性、防误触的工作台，与门店订单状态完美流转。
*   **后台数据驱动**：提供可视化大屏，支持复杂的 SKU 规格、组合套餐配置以及资金审计流水。
*   **高可用系统设计**：利用现代中间件规避高并发超卖，并实现延时任务订单超时自动化管控。

## 2. 系统技术方案与选型

系统整体采用前后端分离架构，通过微服务组件级联与数据库物理持久化结合实现数据高一致性。

```
┌──────────────────────────────────────────────────────────────┐
│                       顾客端 / 骑手端 / 管理端                │ (Vue 3 + Vite + TailwindCSS + Vant/Element)
└──────────────────────────────┬───────────────────────────────┘
                               │ HTTP / SSE
                               ▼
┌──────────────────────────────────────────────────────────────┐
│                    Spring Security 鉴权与 RBAC                │ (Spring Security + JWT)
└──────────────────────────────┬───────────────────────────────┘
                               │
                               ▼
┌──────────────────────────────────────────────────────────────┐
│                      Spring Boot 核心业务层                   │ (MyBatis-Plus + Java 21)
└──────────────────────┬───────────────┬───────────────┬───────┘
                       │               │               │
                       ▼               ▼               ▼
┌──────────────────────────┐     ┌───────────┐     ┌───────────┐
│     MySQL 8.x 持久层      │     │   Redis   │     │ RabbitMQ  │
│ (订单/用户/库存/流水审计)  │     │ (秒杀预扣) │     │(延迟死信) │
└──────────────────────────┘     └───────────┘     └───────────┘
```

| 维度 | 选型组件 | 选用理由 |
| :--- | :--- | :--- |
| **基础语言与容器** | Java 21 + Spring Boot 3.5 | 拥抱虚拟线程与新特性，确保高频计算的极佳吞吐性与现代生态兼容性。 |
| **核心数据库** | MySQL 8.x | 关系型数据库，事务支持强，保障资金流水与订单状态强一致性。 |
| **高速缓存** | Redis 6.x / Lettuce | 存储 JWT 登录状态，并配合 **Lua 脚本** 提供秒杀级高并发 SKU 原子防超卖锁。 |
| **消息队列** | RabbitMQ | 用于削峰解耦，基于 **TTL 与死信队列 (DLX)** 实现订单超时 15 分钟自动撤单。 |
| **持久层框架** | MyBatis-Plus 3.5 | 敏捷编写 SQL，提供逻辑删除（`isDeleted`）、字段拦截填充等高级机制。 |
| **三方 AI 服务** | 火山引擎 DeepSeek-V3 | 接入低延迟大语言模型，并结合 **SSE (Server-Sent Events)** 实现打字机客服流式回复。 |

## 3. 业务功能模块实现总结

### 3.1 顾客端 (Customer)
*   **自助领券与下单**：首页可一键申领特定优惠券（如“新店开业满50减25”）。
*   **智能购物车与多重抵扣**：在购物车模块，结算前动态向后端读取当前用户持有的未消费优惠券，根据门槛金额（`minAmount`）与抵扣规则（满减/折扣/无门槛）在面板上动态置灰或高亮，并在提交时在 Vant `SubmitBar` 组件上展示折后最终支付金额。
*   **订单详情页路由**：顾客下单后，系统抛弃了跳转回“订单列表”的生硬做法，直接返回加密 `orderNo`，并平滑路由至新设计的 `OrderDetail.vue` 详情跟踪页。
*   **双模式微信/余额结算**：详情页集成 Mock 微信支付沙箱与余额扣减扣减，并在回调成功后实时转为“待接单”。

### 3.2 骑手端 (Rider)
*   **暗黑机能风户外工作台**：采用高对比度 Neo-Brutalism 暗黑视觉设计，解决了骑手日常配送时强光反射看不清字体的痛点。
*   **双段滑动确认（Double Swipe）**：接单后，骑手必须通过拖拽“防误触滑动按钮”来流转状态（抢单 -> 取餐 -> 送达），有效规避骑手骑行过程中的误点误碰。
*   **实时流水分账**：骑手送达订单后，配送费直接转入骑手钱包，并在 `user_balance_log` 表中插入变动流水。

### 3.3 管理端 (Admin)
*   **商品级联编辑与 OSS 托管**：可配置商品多规格分类，一键直传商品图至阿里云 OSS。
*   **定制套餐系统**：支持将饮品、蛋糕、多品类规格组合成特色套餐（Combo）并进行营销推送。
*   **全站运力发货中枢**：支持查看门店全部订单流转状态并一键通知骑手抢单派送。

## 4. 关键核心技术细节设计

### 4.1 Redis 原子秒杀库存扣减 (Lua)
为了在高并发下单（如烘焙限时促销）时防止超卖，我们把传统查库、改库的判断逻辑下推至 Redis 内存层，使用 Lua 脚本保证操作的原子性：
```lua
local stock = tonumber(redis.call('get', KEYS[1]))
local num = tonumber(ARGV[1])
if not stock then
    return -2 -- Redis 中无此 SKU 缓存，返回状态码
end
if stock < num then
    return -1 -- 库存不足
else
    redis.call('decrby', KEYS[1], num)
    return 1 -- 预扣减成功
end
```

### 4.2 RabbitMQ 死信超时处理设计
在用户下单时，会产生一条包含订单号的消息投递至 `bakery.ttl.queue`。
*   **死信路由转发**：该队列设置了过期时间 `15分钟`。当消息在队列中静默超时且未被手动消费时，会自动转发至死信交换机 `bakery.dlx.exchange`，被死信消费者处理。
*   **状态与库存原路退回**：死信消费者会查验该订单是否完成了支付，如仍为 `0-未支付`，则执行事务回滚：
    1. 修改订单状态为 `4-已取消`。
    2. 对原有的订单明细 SKU 数量，将原扣减值加回 MySQL 及 Redis 缓存。
    3. 如果该订单使用了优惠券，将 `user_coupon` 的 status 改回 `0-未使用`。

## 5. 项目成效与反思

### 5.1 研发成效
*   **全链路可用性高**：前后端完全调通，从“领券”到“付款”再到“接单配送”，全生命周期状态机可以自动、准确地流动。
*   **开发规范性强**：代码层面采用了强约束的 DTO 输入校验与全局异常捕获，编写单元测试和 Mock 路由时无任何编译警告。
*   **工程构建无瑕疵**：使用 `mvnw clean compile` 及本地环境能够一次性构建打包，容器化提供了最简的 Docker Compose 落地方式。

### 5.2 后续演进建议
*   **地图位置跟踪**：未来可接入高德或腾讯地图 SDK，为骑手提供实时路径规划，为顾客提供地图查看骑手实时 GPS 位置的能力。
*   **真实支付网关集成**：在生产部署时，使用真正的微信支付 V3 API 代替当前的沙箱模拟。
*   **分布式事务**：如果系统在后期扩充为微服务架构，可引入 Seata 来管理多服务（如订单服务与营销券服务）之间的数据强一致性。
